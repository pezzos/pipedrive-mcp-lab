## Objective and acceptance

Build a local, production-oriented remote MCP foundation for one client on Cloudflare Workers. Cloudflare Access authenticates users; Pipedrive OAuth connects the shared CRM tenant; each Access user starts read-only and may change only their own write, delete, and Mailbox permissions. Dry-run remains the default, the model cannot change permissions, and logs persist metadata only.

Acceptance requires the existing stdio/Desktop/plugin behavior to remain compatible; fail-closed Access JWT validation; isolated per-user policy; an independent tenant OAuth broker with encrypted tokens, one-shot state, and serialized refresh; admin-only Pipedrive connection; an accessible CSRF-protected settings UI; correct Mailbox/delete/dry-run gates; redacted audit events; a Worker-compatible bundle without Node built-ins; local MCP HTTP and OAuth tests using fakes; `npm run check`; local Worker validation; and an exact sandbox-to-production runbook.

## Settled user decisions

- Single-tenant remote MCP for one client, hosted on Cloudflare Workers.
- Cloudflare Access Managed OAuth for user identity.
- Pipedrive OAuth for the shared CRM account; one administrator authorizes it once, then refresh tokens sustain access.
- Permissions are stored per Access user. Each allowlisted user may self-enable their own writes, deletes, and Mailbox capability without redeployment.
- Read-only and `dry_run=true` are defaults. Deletes and Mailbox are separate toggles.
- No CRM payload persistence in logs; only operational metadata and stable pseudonymous identity.
- Validate first against a Pipedrive Developer Sandbox, then promote the same artifact/configuration shape to production.

## Architecture decision

The Worker stays stateless per HTTP request and instantiates the MCP server with injected runtime diagnostics. A `UserPolicy` Durable Object is keyed by Access `sub` and stores only that user's switches. A separate singleton `TenantSecrets` Durable Object stores encrypted Pipedrive OAuth material, one-shot states, and serialized refresh state. Policy reads therefore never wait on an OAuth network operation. The Pipedrive connection route is restricted to `REMOTE_ADMIN_EMAIL`; ordinary users can modify only their own policy.
